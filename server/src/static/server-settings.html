Series Generators

How generators work

Each generator has two scripts that share the same state object:

  • Init script — runs once when the generator is saved and once again when the server starts. Use it to initialise stateful objects (e.g. a phase accumulator) and store them in state.
  • Process script — runs at the configured interval. This is where you produce data and push it via ingest().

Unlike processors, generators have no input trigger — they produce data autonomously. When a generator calls ingest(), the new data flows through the store and automatically triggers any matching processors, just like HTTP or MQTT ingestion.

The process script can call ingest() with any number of points per tick, not just one. Producing a batch of N points per tick (with timestamps spread over the interval) is the recommended pattern: it keeps series smooth even at low generator rates and is more efficient than one point per tick at very high rates.

The state object persists across ticks and survives script edits (cleared only on server restart or explicit "Save & Clear state"). The init script re-runs on every save and can repopulate specific keys while leaving others intact.

Both scripts execute inside a secure VM sandbox with a 5-second timeout. Errors are caught and shown as red badges in the list; they do not stop the generator's interval from continuing.

Example — Sine-wave generator (init + process scripts)

Configure with Interval: 100 ms. This produces 5 samples per tick at 50 Hz, yielding a smooth 1 Hz sine of amplitude 1. Producing a batch of points per tick keeps the waveform smooth regardless of the generator rate.

Init script

state.phaseRad = 0;
state.lastTs   = null;

Process script

// Parameters — adjust to taste
const FREQ_HZ     = 1;     // signal frequency in Hz
const AMPLITUDE   = 1;     // peak amplitude
const SAMPLE_HZ   = 50;    // samples per second to produce each tick
const intervalSec = 0.1;   // must match the configured Interval (100 ms)

const now    = Date.now();
const stepMs = 1000 / SAMPLE_HZ;
const count  = Math.round(intervalSec * SAMPLE_HZ);

const timestamps = [];
const values     = [];

for (let i = 0; i < count; i++) {
  const ts    = now - (count - 1 - i) * stepMs; // anchor batch to wall-clock
  const dtSec = state.lastTs == null ? 0 : (ts - state.lastTs) / 1000;
  state.phaseRad += 2 * Math.PI * FREQ_HZ * dtSec;
  state.lastTs    = ts;
  timestamps.push(ts);
  values.push(+(AMPLITUDE * Math.sin(state.phaseRad)).toFixed(6));
}

ingest({
  measurementName: "sine_gen",
  points: { timestamps, series: { value: values } }
});

    Derived Series Processors

    How processors work

    Each processor has two scripts that share the same state object and sandbox API:

    • Init script — runs once when the processor is saved and once again when the server starts. Use it to create stateful objects (e.g. an FFT analyser or EMA filter) and store them in state.
    • Process script — runs after every ingest (HTTP or MQTT). This is where you read measurements, apply transforms, and write derived data back via ingest().

    Both scripts execute inside a secure VM sandbox with a 5-second timeout. Errors are caught, logged to the server console, and shown as red badges in the list (separately for init and process errors). They do not stop other processors from running.

    A processor that calls ingest() to write derived data will not trigger another processor sweep (re-entrancy guard), so there is no risk of infinite loops.

    The state object persists across process-script runs for the lifetime of the server process (resets on restart). Editing and saving a processor preserves state — the object is not reset when you update the scripts. When you save, the init script re-runs and can overwrite specific keys (e.g. state.sp = dsp.spectrum({...})) while leaving others intact. A full server restart clears everything and re-runs init from scratch.

    Example — EMA using init + process scripts

    Init script (runs once on save / server start)

    // Create a stateful EMA filter with alpha = 0.1.
const dsp = require('dsp');
state.ema = dsp.ema(0.1);

    Process script (runs on every ingest)

    const m = getMeasurement("cpu_like");
if (!m || getNewPointCount("cpu_like") === 0) return;

const count = getNewPointCount("cpu_like");
const start = m.timestamps.length - count;
const newTs  = m.timestamps.slice(start);
const newEma = state.ema.update(m.series.value.slice(start));

ingest({
  measurementName: "cpu_like",
  points: { timestamps: newTs, series: { ema: newEma } }
});

    Loading the DSP library

    Use the built-in require() function at the top of your processor script: 

    const dsp = require('dsp');

    require() is a restricted sandbox function — it only exposes approved built-in modules. It does not access the filesystem or node_modules.

    dsp.ema(alpha) — Exponential Moving Average

    Creates a stateful EMA filter instance. Store it in state so the accumulator persists between processor runs.

    NameTypeDescription
    dsp.ema(alpha)constructorReturns a new EMA instance. alpha must be in (0, 1). Higher = faster response. Lower = more smoothing. Time constant ≈ 1 / alpha samples.
    instance.update(values)functionFeed new values. Returns a number[] of the same length with smoothed output.
    instance.reset()functionClear the accumulator (useful after a data gap).
    instance.valuegetterCurrent accumulator value (last output), or undefined before first update.

    dsp.median(windowSize) — Sliding-Window Median Filter

    Creates a stateful median filter instance. Store it in state so the sliding window persists between processor runs. Effective at removing impulse noise (spikes) while preserving sharp edges — unlike EMA, it can reject single outliers completely.

    NameTypeDescription
    dsp.median(windowSize)constructorReturns a new median-filter instance. windowSize must be a positive integer ≥ 1. Larger windows give stronger noise rejection but more lag.
    instance.update(values)functionFeed new values. Returns a number[] of the same length. Each output element is the median of the last windowSize inputs seen so far.
    instance.reset()functionClear the internal sliding window (useful after a data gap).
    instance.buffergetterCurrent contents of the sliding window as a number[].

    dsp.spectrum(options?) — Spectrum Analyser

    Creates a stateful FFT analyser instance. Store it in state and call compute(values) on each new batch. The analyser applies a configurable window function, optional dB scaling, and per-bin exponential spectral averaging to reduce frame-to-frame noise.

    OptionTypeDefaultDescription
    sampleRateHznumber1Sample rate in Hz. Omit to get normalised frequencies in [0, 0.5].
    windowstring'rect'Window function applied before the FFT to reduce spectral leakage. One of: 'rect' (no window), 'hann', 'hamming', 'blackman'. 'hann' is a good general-purpose choice.
    scalestring'magnitude'Output scale: 'magnitude' (one-sided normalised amplitude) or 'db' (20·log10 of the normalised amplitude; 0 dB = amplitude 1, so a unit sinusoid reads 0 dB). Zero-magnitude bins are clamped to −120 dB so all output values remain finite.
    averagingnumber1Number of frames used for EMA spectral averaging. 1 = no averaging (raw spectrum). Higher values give a smoother spectrum that reacts more slowly. The EMA coefficient is 2 / (averaging + 1).
    Method / propertyDescription
    instance.compute(values)Process a new frame. Returns { frequencies, magnitudes }. Magnitudes are window-normalised: a real sinusoid of amplitude A yields a peak of A at its positive-frequency bin (one-sided convention). Input is auto zero-padded to the next power of 2. If the input length changes between calls the accumulator is reset automatically.
    instance.configure(opts)Change any option at runtime (same keys as the constructor). Only supplied keys are updated; the rest keep their current value. Resets the accumulator.
    instance.reset()Clear the spectral averager history (useful after a data gap or signal change).
    → .frequenciesnumber[] — frequency of each bin in Hz (DC to Nyquist).
    → .magnitudesnumber[] — amplitude (or dB) at each bin, after averaging.

    Amplitude normalisation

    Magnitudes are divided by half the sum of the window coefficients (windowSum/2), making the output independent of the block size N and of the window type used. The convention is one-sided: a real sinusoid of amplitude A yields a peak of exactly A at its positive-frequency bin. For scale: 'db', 0 dB corresponds to amplitude 1 — a unit sinusoid therefore reads 0 dB at its peak bin.

    Tip: compute sampleRateHz from timestamps (ms) as 1000 / (timestamps[1] - timestamps[0]). Use power-of-2 block sizes (128, 256, 512 … samples) for best efficiency. For live streaming, averaging: 8 with window: 'hann' is a good starting point.

    Example — EMA using the DSP library

    const dsp = require('dsp');
const m = getMeasurement("cpu_like");
if (!m || getNewPointCount("cpu_like") === 0) return;

if (!state.ema) state.ema = dsp.ema(0.1);

const count  = getNewPointCount("cpu_like");
const start  = m.timestamps.length - count;
const newTs  = m.timestamps.slice(start);
const newEma = state.ema.update(m.series.value.slice(start));

ingest({
  measurementName: "cpu_like",
  points: { timestamps: newTs, series: { ema: newEma } }
});

    Example — Spectrum with Hann window, dB scale, and spectral averaging

    Init script

    // Create the spectrum analyser once; it will be re-created if you change these options and save.
const dsp = require('dsp');
const m = getMeasurement("cpu_like");
const dt = m ? (m.timestamps[1] - m.timestamps[0]) : 100; // fallback 100 ms
state.sp = dsp.spectrum({
  sampleRateHz: 1000 / dt,
  window:       'hann',
  scale:        'db',
  averaging:    8
});

    Process script

    const m = getMeasurement("cpu_like");
if (!m || m.timestamps.length < 2) return;

// Feed the last 256 samples on every run
const N = 256;
const { frequencies, magnitudes } = state.sp.compute(m.series.value.slice(-N));

// Use frequency values as pseudo-timestamps so the x-axis shows Hz
const now = Date.now();
ingest({
  measurementName: "cpu_like_spectrum",
  points: {
    timestamps: frequencies.map((_, i) => now + i),
    series: { dB: magnitudes }
  }
});
